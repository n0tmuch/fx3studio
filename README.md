# Fx3 Studio

Portfolio site for Spencer Harrison: fashion designer, BFA Otis College of Art and Design.

Live: https://fx3studio.com (also reachable at https://fx3studio.vercel.app)

## Stack

- Vite 6 + React 18 with **SSG + hydrateRoot** (Phase 6): every public route is server-rendered at build time, then hydrated client-side
- Vanilla CSS (no framework)
- `<image-slot>` web component for fillable image placeholders (kept for future authoring; not in the rendered tree of any of the 12 SSG'd routes)
- Hosted on Vercel; auto-deploys from `main` (GitHub: [n0tmuch/fx3studio](https://github.com/n0tmuch/fx3studio))

## Develop

```bash
npm install
npm run dev      # http://localhost:5173
npm run build    # production build → dist/
npm run preview  # serve dist/ locally
```

Pushing to `main` triggers a Vercel production deploy. Pushing any other branch creates a preview deployment (gated by Vercel SSO).

## Project layout

```
index.html              Vite entry for dev (`npm run dev`). Production routes are generated by the SSG step; this file is overwritten in dist/.
vercel.json             explicit framework: vite, plus SPA rewrites for any /work/<slug> that misses a prerendered file
src/
  main.jsx              React client root: hydrateRoot on top of SSG'd HTML, with createRoot fallback for dev
  App.jsx               top-level page; accepts initialRoute prop so SSR and first client render produce the same tree
  Components.jsx        Nav, Hero, About, WorkGrid, CV, Contact, ProjectDetail, FifaCaseStudy, Slot, etc.
  data.js               site copy + collection metadata: edit for content changes
  styles.css            all styles (cinema theme via [data-theme])
  entry-server.jsx      SSR entry. render(url) → { html, initialRoute } via react-dom/server.renderToString
  case-studies/         one file per non-FIFA collection (Bloodmoon, Frame, Gothic, ...)
  lib/
    image-slot.js       <image-slot> custom element (kept for authoring tools; not used in any SSG'd route)
    tweaks.jsx          stub for the design-tool tweaks panel (renders nothing)
scripts/
  prerender.mjs         post-build: renders the React tree per route, writes static HTML for /, /work/<slug> × 10, /cv
public/
  assets/               runway.mp4 + per-collection imagery
  robots.txt            allow all + sitemap pointer
  sitemap.xml           home + all 10 /work/<slug> URLs + /cv
  favicon.svg           placeholder serif "fx3" mark
```

The build pipeline is `npm run build` → `vite build` (client bundle: `dist/assets/index-<hash>.{js,css}`) → `vite build --ssr src/entry-server.jsx --outDir .ssr-build` (Node-runnable SSR bundle) → `node scripts/prerender.mjs` (imports the SSR bundle, calls `render(url)` for each of the 12 public routes, wraps the rendered HTML in a per-route page template with title / meta / OG / JSON-LD, writes `dist/index.html`, `dist/work/<slug>/index.html` × 10, and `dist/cv/index.html`). Each page embeds `window.__INITIAL_ROUTE__ = "<slug>|null"`. On load, `main.jsx::hydrateRoot` reads it and passes it to `<App initialRoute=…>`, so the server tree and the client's first render match exactly. Tier-one crawlers and tier-two AI/recruiter extractors (the latter apply CSS) see the same DOM real browsers paint — no hidden shell, no parallel content.

Content edits almost always live in `src/data.js`. The `COLLECTIONS` array drives the work grid and case-study dispatch in `Components.jsx::ProjectDetail`.

### 2026-05-12 (still later): prerender visibility fix

External AI fetchers (Claude `web_fetch`, Perplexity, link-preview cards) were still reporting "site has no body content" *after* Phase 3 shipped. The prerendered HTML was reaching them — `curl -A "PerplexityBot" https://fx3studio.com/` returns 17 KB with full H1/H2/H3 hierarchy — but cheap content extractors were skipping the body because Phase 3's shell was wrapped in inline `<div style="position:absolute;clip:rect(0 0 0 0);width:1px;height:1px;overflow:hidden">`. That is the canonical "sr-only" pattern; HTML parsers that don't run a full layout engine still detect it as inline-hidden and drop the subtree. The cloak we put up for human FOUC also cloaked the content from the very consumers Phase 3 was built for.

**Fix (commit `65d7cca`):**
- `scripts/prerender.mjs`: replaced the inline `style="${SR}"` wrapper with `class="prerender-shell"` on both `homeShell()` and `projectShell()`. The `SR` constant is gone; the rationale comment now documents *why* class-based hiding is the right move for this consumer mix.
- `src/styles.css`: added `.prerender-shell { display: none; }` directly after the `body` baseline (line 68). The class lives in the bundled stylesheet, not inline, so a raw-HTML reader that doesn't fetch + parse external CSS sees the content; a real browser blocks first paint on the stylesheet and hides the shell cleanly before React mounts. `main.jsx` uses `createRoot` (not `hydrateRoot`), so React replaces `#root` wholesale on mount — no hydration mismatch concern.

**Verification:**
- `curl -sS https://fx3studio.com/ | grep -c "clip:rect"` → `0` (was `1` per shell on all 12 prerendered pages)
- `curl -sS https://fx3studio.com/ | grep -oE '<h[1-3][^>]*>[^<]+</h[1-3]>' | wc -l` → 12 headings present in raw HTML, identical to pre-fix output
- `.prerender-shell{display:none}` is bundled into `dist/assets/index-<hash>.css`
- Real-browser load: shell never paints (render-blocking CSS hides it before first frame), React mounts as before

### 2026-05-13 (Phase 6): SSG + hydrateRoot — visible DOM is the real React tree

Round-three crawl audit found that tier-two extractors (Claude `web_fetch`, Perplexity, ChatGPT search, recruiter-side AI auditors) that fetch raw HTML *and apply CSS* were applying the `.prerender-shell { display: none }` rule from Phase 5's visibility-fix round and reporting empty bodies — exactly the consumers Phase 5 was meant to reach. The rule kept the parallel shell invisible to humans and to AI extractors that follow CSS, but the bundled stylesheet always loads in real browsers, so the cloak that protected the visible UX from a FOUC also cloaked the inventory from CSS-aware AI consumers. The cleanest fix is structural: stop maintaining two parallel DOMs and make the visible site itself be what tier-two extractors see.

**The change:** every public route is now server-rendered at build time and hydrated on the client. The SSR'd HTML *is* the visible UI. No hidden parallel shell, no `display: none` cloak, no FOUC: server tree and client first render produce the same HTML, byte-equivalent, and React 18's `hydrateRoot` adopts the existing DOM without re-creating it.

**Architecture (rolled-our-own, no `vite-react-ssg` / Astro / Next migration):**
- `src/entry-server.jsx` is the SSR entry. `render(url)` resolves the route's `initialRoute` (a slug or `null`) and calls `react-dom/server.renderToString(<App initialRoute={…} />)`.
- `src/App.jsx` accepts an `initialRoute` prop. The `openId` `useState` initializer now reads it directly (no `typeof window` branch), so SSR and the first client render produce identical trees.
- `src/main.jsx` switched `createRoot` → `hydrateRoot`. It reads `window.__INITIAL_ROUTE__` (injected by the SSG step into every page) and forwards it to `<App>`. There is a `createRoot` fallback for the dev server and any cache miss that serves an un-prerendered shell.
- `scripts/prerender.mjs` was rewritten. The build is now two passes:
  1. `vite build` produces the client bundle in `dist/`.
  2. `vite build --ssr src/entry-server.jsx --outDir .ssr-build` produces a Node-runnable SSR bundle.
  3. `node scripts/prerender.mjs` imports the SSR bundle, calls `render(url)` for each of the 12 public routes, wraps the rendered body in a per-route template with `title` / `meta` / `OG` / `Twitter` / canonical / JSON-LD `@graph` (Person + Organization + WebSite + the relevant `CreativeWork` nodes, full `image[]` arrays included), embeds `window.__INITIAL_ROUTE__`, and writes the file. Image-count audits per project trip the build if a project's rendered `<img>` count drops below its Phase-5 baseline.
- `src/lib/image-slot.js`'s `customElements.define` is now guarded with `typeof customElements !== 'undefined'` — defensive against future drift (today's SSR bundle doesn't import this file, but if anything reaches for it the Node-side guard keeps the build healthy).
- `src/styles.css`: deleted the `.prerender-shell { display: none }` rule. The class is no longer emitted anywhere — the rule had no targets.
- `package.json::build`: `vite build && vite build --ssr src/entry-server.jsx --outDir .ssr-build && node scripts/prerender.mjs`.
- `.gitignore`: added `.ssr-build/` (intermediate SSR bundle, not shipped).

**Hydration hazards inventoried before the switch:**
- `App.jsx` previously read `window.location.pathname` inside the `useState` initializer — only existed on the client, would mismatch the SSR `null`. **Fix:** pass `initialRoute` in as a prop from both the SSR `render()` call and the client `hydrateRoot` call (reading `window.__INITIAL_ROUTE__`).
- `lib/image-slot.js` calls `customElements.define` at module load. `customElements` is undefined in Node. **Fix:** `typeof customElements !== 'undefined'` guard. (In practice the SSR bundle doesn't import this file, but the guard removes a future foot-gun.)
- `lib/tweaks.jsx` initializes with `useState(defaults)` — pure, deterministic, no browser globals. Safe as-is.
- `Components.jsx::Hero` uses `videoRef.current.play()` in a `useEffect`, runs client-only. `<video autoPlay muted loop playsInline poster=…>` renders identically on both sides; React-DOM-server emits `playsinline=""` `autoplay=""` etc. consistent with hydration expectations.
- `Components.jsx::Nav` scroll listener and `ProjectDetail`'s `document.body.style.overflow` are both `useEffect`-only.
- Theme: `index.html` has `data-theme="cinema"` on `<html>` statically; the `useEffect` only diverges if a tweak changes the theme. No mismatch on first render.
- `new Date().getFullYear()` in the footer: same value (2026) on the SSG build host and on every client clock today. Acceptable risk across the year boundary — React 18 hydration warnings on year don't break rendering.

**Is this materially better than Phase 5 for AI / crawler / recruiter visibility?**

Yes — for the specific class of extractors Phase 5 was meant to reach but missed. Tier-one crawlers (raw HTML scanners that ignore CSS, e.g. Googlebot first pass, LinkedIn unfurl, most ATS scrapers) already saw the inventory under Phase 3–5; they keep seeing it. Tier-two extractors (Claude `web_fetch`, Perplexity, ChatGPT search, recruiter-side AI auditors) that fetch + apply CSS were reporting an empty body under Phase 5 because `.prerender-shell { display: none }` hid the parallel content. Under Phase 6 there is no parallel content — the React tree is the body, served pre-rendered, and there is no `display: none` rule to apply. Verifiable by curl: the same `<img>` inventory and case-study text appears in the raw HTML, but now those same elements are also the visible UI in a real browser (no separate visible-vs-hidden DOM). For the question "does an AI auditor opening this URL see Spencer's portfolio?" the answer is no longer dependent on which CSS rules they choose to apply.
